Manual para desarrollar una página Web accesible para adultos mayores

A la hora de interactuar con páginas Web, a los adultos mayores se les pueden presentar ciertas dificultades y barreras que les impiden aprovechar al máximo lo que estas ofrecen. Refiérace a la sección accesibilidad para adultos mayores de la página de Conceptos para entender más sobre estas dificultades.

El objetivo de esta página es servir de manual o guía para la creación y desarrollo de páginas Web que son accesibles para adultos mayores en su totalidad. Para ello, se utilizan varios recursos presentes en material adicional. El principal es el WCAG 2.1, que son recomendaciones y lineamientos a cumplir para que una página Web sea accesible. Para alcanzar dichas recomendaciones se utilizan además dos herramientas: Accessible Rich Internet Applications (ARIA), que son un conjunto de atributos de HTML que ofrecen información útil a lectores de pantalla y otras tecnologías sobre el contenido de una página que de lo contrario no sería posible de identificar, y también la herramienta WAVE, que sirve para evaluar el nivel de accesibilidad de una página Web existente. Más adelante se entrará en detalle con estas herramientas.

1. Desarrollo Web accesible con WCAG 2.1

Al desarrollar una página Web, lo más importante es tener en mente implementar la accesibilidad desde antes de comenzar a crearla y durante todo su desarrollo. Así es más fácil considerar en todo momento los aspectos que hacen una Web accesible, sin tener que volver a escribir código cuando ya es demasiado tarde. Por cada componente, texto, información o demás que sea agregado a la página se debe pensar: ¿es comprensible? ¿su función es clara? ¿cualquier persona puede identificarlo, entenderlo y/o usarlo? ¿se puede interactuar con cualquier medio (teclado, mouse, pantalla táctil, voz, etc)? ¿su contexto y posición respecto al resto de la página es adecuado? ¿es necesario?

Web Content Accessibility Guidelines 2.1 o WCAG 2.1 son las actuales recomendaciones dadas por el World Wide Web Consortium (W3C) para que una página Web sea accesible. Se pueden encontrar 4 principios que toda página web debe tener para considerarse accesible, que son:

  • Perceptible: Los componentes de la interfaz de usuario deben ser mostrados a los usuarios en formas que ellos puedan entender.
  • Operable: Los componentes de la interfaz de usuario y la navegación debe ser manejable.
  • Comprensible: La información y las operaciones de los usuarios deben ser comprensibles.
  • Robusto: El contenido debe ser suficientemente robusto para que pueda ser interpretado por una variedad de agentes y tecnologías asistivas.

A continuación se mostrarán los puntos más importantes incluidos en estos 4 principios, sus técnicas y recomendaciones así como también errores que no se deben cometer.

1.1. Perceptible

Todos los componentes de una página Web deben poder ser identificados y entendidos por todos los usuarios. Muchos adultos tienen dificultades con la visión, y letras demasiado pequeñas, colores con bajo contraste y demás factores dificultan su percepción de las páginas. Algunos utilizan lectores de pantalla para percibir la información, por lo que no es posible concentrarse únicamente en el diseño visual de la página. Además, otros tienen dificultades auditivas, por lo que demostrar información únicamente por audio o videos sin subtítulos no es viable.

1.1.1. Alternativas textuales

Todo contenido de una página que no se basa en texto debe tener alternativas textuales que pueden ser traducidos por el medio de elección del usuario. El ejemplo más claro de esto es en el uso de imágenes. Toda imagen incluida en una página debe ofrecer un texto alternativo para que usuarios que no puedan verla puedan comprender su contenido. Con HTML, el elemento img, usado para introducir imagenes a una página, contiene el atributo alt. El texto ingresado a dicho atributo servirá de texto alternativo para la imagen.

También pueden existir algunos componentes que, con solo el tipo de elemento y contexto en la página, además de no tener atributos adicionales de texto alternativo, no logran mostrar su función a usuarios que no pueden verlos. Un ejemplo de esto es una scrollbar personalizada cuyo elemento es un div. Para ello se pueden utilizar atributos ARIA, como el atributo role para declarar el rol de dicho elemento, o el atributo aria-label para agregar un texto que servirá para describir la función o significado del elemento. En la sección 2. Atributos y Semántica ARIA se explica de mejor forma el uso de ARIA para desarrollar páginas Web.

1.1.2. Multimedia

En casos en que se desee agregar un audio o un video a la página, estos deben venir con algún tipo de alternativa para personas que no pueden escuchar o ver dicho contenido. Si solo se trata de audio, debe existir algún tipo de transcripción o subtítulos en la página o un enlace a ello. Si se trata de un video, este debe venir con subtítulos claros y descriptivos de lo que se dice y suena en el video, y también una transcripción o audio descriptivo de lo que se ve y escucha en dicho video. En caso de ser una transmisión en vivo, es recomendable implementar alguna tecnología que transcriba automáticamente su contenido.

1.1.3. Adaptabilidad y propósito

No todos los usuarios van interactuar con una página de la misma forma. Unos pueden usar lectores de pantalla, otros pueden cambiar el tamaño del contenido y otros pueden simplificar el diseño con ayuda de alguna tecnología. Es importante que la página sea accesible para percibirla de cualquier forma y se adapte a todo sin perder contenido, cambiar su propósito o cambiar su orden.

Si el diseño visual de la página, su orden y secuencia es importante para comprender correctamente la información de la misma, debe ser posible entenderla de igual forma sin la necesidad de verla como se diseña originalmente. Es importante hacer un muy buen uso de HTML, sus elementos, atributos, orden y estructura para que esto sea posible.

Debe ser posible aumentar el tamaño de la página o cambiar su orientación sin que se pierda información de la misma o sea muy confusa. La página debe ser sensible a adaptarse a cualquier dispositivo y tamaño. Varias prácticas de HTML permiten hacer una página Web "responsive". También existen herramientas útiles que facilitan diseñar páginas de esta forma, como por ejemplo Bootstrap.

En la medida de lo posible, se deben usar los elementos por defecto de HTML según su propósito y estructura. Por ejemplo, usar h1 para headers principales, usar main para indicar el contenido principal de la página, usar nav para identificar algún tipo de barra de navegación en la página, y demás. Además, es importante ordenar estos elementos de manera que la estructura no sea confusa y que el DOM siga el diseño visual de la página. En caso de que no exista un elemento de HTML que identifique correctamente algún contenido personalizado, es posible usar atributos ARIA para identificar correctamente el rol que cumple dicho elemento. El objetivo de todo esto es para que tecnologías asistivas logren comprender correctamente la estructura de la página y el propósito de todos sus elementos para que no exista ninguna confusión.

Por último, no se debe evidenciar el propósito de algún elemento únciamente de manera visual. Un ejemplo de esto es poner un fondo de color rojo a un mensaje de error, sin explicar de ninguna otra manera que es un error. Es importante agregar texto adicional o algún otro mecanismo que sirva para identificar el significado del elemento. En el caso del ejemplo anterior, basta con poner "ERROR" o algún texto similar en el mensaje.

1.1.4. Visibilidad

Las dificultades visuales en adultos mayores no tratan solo de ceguera completa. Existen personas con visión deficiente, daltonismo, percepción reducida y demás. Estas personas sí llegan a utilizar las páginas de manera visual, pero necesitan que sean fáciles de ver y entender.

El uso de colores debe ser muy cuidadoso. El contraste entre colores debe ser suficientemente alto para que todo tipo de persona pueda diferenciarlos. Al agregar texto, este debe tener color que haga un gran contraste con respecto al color de su fondo, de lo contrario algunas personas tendrán dificultades en leerlo. El Contrast Checker de WebAIM es una útil herramienta que evalúa el contraste de colores para que sean accesibles.

Siguiendo con el tema del texto, este debe ser debidamente legible. Se deben usar tipografías que sean fáciles de leer, y en la medida de lo posible, no usar texto demasiado pequeño. Además, debe ser posible agrandar este texto en el navegador sin que exista una pérdida en el contenido y estructura de la página. Por último, como recomendaciones adicionales, el texto debe tener una separación entre palabras, líneas y párrafos lo suficientemente amplia para que no sea confuso para ciertas personas.

Por último, en la medida de lo posible, no se deben usar imagenes que solo contienen texto. En estos casos es mejor escribir el texto directamente en la página para que pueda alterarse de la forma que el usuario desee.

1.2. Operable

Todos los elementos, componentes, formularios, navegación y demás deben poderse usar por todas las personas. No todos interactúan con la página de la misma forma, ni usan los mismos dispositivos de entrada. Algunos solo usan teclado, otros utilizan la voz y otros pantallas táctiles. También, todos deben poder entender cómo utilizar la página, ubicarse en ella y navegar sin confusión ni problema alguno.

Adicionalmente, es importante considerar que personas con trastornos como la epilepsia van a interactuar con la página y esta no debe atentar contra su salud. No se deben utilizar secuencias de imágenes rápidas, luces parpadeantes o animaciones rápidas que puedan causar reacciones físicas peligrosas en estas personas.

1.2.1. Accesibilidad con el teclado

Muchos adultos mayores no tienen la destreza suficiente para usar el mouse, o no lo usan por problemas de visión, o en general prefieren utilizar únicamente el teclado como dispositivo de entrada. Las páginas Web accesibles deben permitir interactuar con ella en su totalidad únicamente a través de un teclado. En general, esto puede alcanzarse con un buen diseño con HTML y la estructura del DOM, utilizando los debidos elementos en cada caso.

Además, no deben existir "keyboard traps", que son elementos o componentes que, al recibir el enfoque del teclado, no tienen ninguna forma de cambiar dicho enfoque a través de únicamente el uso del teclado.

1.2.2. Tiempo

Algunas páginas tendrán algún tiempo límite para percibir un contenido o para llenar un formulario. En la medida de lo posible, no se debe limitar el tiempo en que un usuario puede usar un componente de la página, ya que muchos adultos mayores necesitan tomarse su tiempo para utilizarlo. En caso de que el tiempo límite sea necesario, el usuario debe saber que existe y tener la posibilidad de desactivarlo o extenderlo. Cuando no es posible cambiar este tiempo límite, como en un timeout en la compra de un producto, como mínimo se debe comunicar de forma clara al usuario de dicho tiempo límite y si implica alguna périda de datos en la sesión.

1.2.3. Navegación e input

Todos los usuarios deben tener la posibilidad de navegar en la página de manera sencilla y comprensible, sin importar la forma en que lo hagan. También, todo tipo de dispositivo de entrada debe ser aceptado.

Para usuarios que no usan el mouse para navegar, una técnica bastante útil es agregar un botón u opción para navegar directamente al contenido u otra sección importante de la página, sin necesidad de recorrer por componentes como la barra de navegación o el encabezado. Puede hacer la prueba con esta misma página, si al ingresar a ella presiona tabulador podrá acceder a botones ocultos para navegar por la misma (es necesario que el enfoque del telcado esté en el inicio de la página). Además, cuando no se utiliza mouse el orden del enfoque del dispositivo de entrada debe tener una secuencia con sentido. No debe saltar entre componentes distantes ni en orden regresivo.

El texto de la página debe ayudar a saber en qué lugar de ella se encuentra y en general proporcionar buena navegabilidad. Se deben usar elementos de headings con sentido para encabezados o títulos, y estos deben tener un texto que logren identificar correctamente la sección de dicho encabezado. Además, enlaces e hipervínculos deben tener un texto que identifiquen claramente por sí solos hacia dónde se dirigen.

En páginas que contengas muchas secciones, aportar una tabla de contenidos con enlaces a todas sus secciones es muy útil para una navegabilidad óptima.

1.3. Comprensible

Todo usuario debe tener la posibilidad de entender correctamente el contenido de la página Web, incluyendo el texto y componentes de la interfaz.

1.3.1. Legibilidad

Primeramente, la página en general así como también su contenido deben tener identificados el lenguaje en el que están escritos. Así, tecnologías que asisten al usuario para percibir una página saben en qué idioma deben reproducirla. Para ello se puede usar el atributo lang de HTML para identificar el idioma de la página o el contenido. En caso de que sea posible que usuarios de otros idiomas interactúen con la página, es útil que exista la opción de cambiar de idioma.

También en casos donde se usen palabras o abreviaturas desconocidas es recomendable adjuntas alguna descripción de texto antes o después que explique su significado. Cuando las palabras tengan una pronunciación ambigua también se recomienda escribir dicha pronunciación.

1.3.2. Predictabilidad

Las páginas Web deben aparecer y operar en formas predecibles para todos los usuarios. La idea es que la interacción la página no sea confusa y se asemeje a experiencas pasadas con otras páginas. La estructura de la página es vital para esto, y dar un buen orden a los elementos y funciones claras ayudan al usuario a comprender cómo usar la página.

Un cambio de contexto, o cambios en la página que alteran la forma en la que se pueda percibir o la información en ella, no deben suceder de manera impredecible. Darle el enfoque a un elemento o algún input no deben cambiar el contexto sin antes avisarle al usuario de dicha interacción.

Además, en un sitio Web con varias páginas, navegar entre ellas debe ser predecible. Todas las páginas deben seguir elementos y mecanismos de navegación consistentes en el mismo orden, por ejemplo proporcionar una barra de navegación en todas las páginas. También, elementos que se repiten en varias páginas deben identificarse de la misma manera y cumplir la misma función.

1.3.3. Asistencia en la interacción

Se deben proporcionar ayudas para que los usuarios logren interactuar correctamente con el sistema, principalmente en caso de que tengan que llenar algún formulario.

Si ocurre algún error con la entrada que ingresó un usuario, como correo electrónico no existente, el sistema debe avisar claramente y textualmente al usuario del error y de cómo evitarlo. Se pueden utilizar atributos de ARIA para identificar de mejor forma qué debe ir en cada entrada de un formulario, así como también identificar que campos son requeridos.

En general, cuando se necesite de la interacción y entradas por parte de un usuario se deben proporcionar instrucciones suficientemente claras para que no exista ningún tipo de error, y el sistema debe captar excepciones cuando los datos ingresados no son correctos.

1.4. Robusto

Este último principio es bastante directo y simple. Toda página Web accesible debe ser robusta y compatible para poder se interpretada por cualquier agente, como tecnologías de asistencia. En general, para que una página basta con que sea programada de la mejor manera, ayudándose de todo lo que HTML y ARIA pueden proporcionar.

Se debe programar el HTML siguiendo la especificación y los estándares del mismo. Un recurso muy útil con la documentación de HTML es la Referencia HTML de MDN.

Además, como se ha indicado anteriormente, todos los elementos deben tener bien definido su rol, función y/o nombre. Esto puede lograrse con usar los elementos correctos de HTML, y si con esto no basta, se pueden utilizar roles y labels de ARIA, para dejar en claro todos estos datos para lectores de pantalla y otras tecnologías.

2. Atributos y semántica ARIA

Accesible Rich Internet Applications (ARIA) es un framework que provee un conjunto de herramientas y atributos que mejoran el nivel de accesibilidad de las páginas Web. Se pueden especificar diversos atributos a elementos HTML que modifican la accesibilidad de la página, agregando características reconocibles por tecnología de asistencia como lectores de pantalla.
